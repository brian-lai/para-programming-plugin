PARA-Programming

Structured AI-Assisted Development

A methodology for professional software engineering with AI coding assistants

What is PARA-Programming?

PARA-Programming combines three proven methodologies:

  • PARA Method (Tiago Forte) - Organizational framework
  • Pair Programming - Collaborative development for human-AI teams
  • Persistent Context - Token-efficient memory across sessions

Goal: Make AI-assisted development auditable, repeatable, structured, and efficient.

Diagram: Human + AI + Context Directory + Tools = Second Brain

The Problem: Current State of AI-Assisted Development

Working with agentic AI tools introduces challenges:

  1. Auditability - "What happened? When? Why?"
  2. Replayability - "How do I continue where I left off?"
  3. Structure - "Not all work is linear"
  4. Context Window Limitations - "Token limits force trade-offs"

Pain Point #1: Auditability

The Problem: "What happened? When? Why?"

  • No clear record of AI's reasoning
  • Code commits lack decision context
  • Difficult to review AI changes weeks later
  • Compliance requirements can't be met
Real-World Scenario:
Developer: "Claude, why did we choose Redis over Memcached?"
Claude: "I don't have context from previous sessions."

The session that made that decision is gone.
Timeline: Session 1 → Session 2 → Session 3 (context lost)

Pain Point #2: Replayability

The Problem: "How do I continue where I left off?"

  • Sessions are isolated - closing loses all context
  • New sessions start from zero
  • Can't hand off work between team members
  • Can't resume after interruptions
Real-World Scenario:
Friday 5pm: Complete complex refactoring with Claude → Close laptop
Monday 9am: Open new session → "What were we working on?"
"I don't have context from previous conversations."
Diagram: Sessions as isolated bubbles with no connections

Pain Point #3: Structure

The Problem: "Not all work is linear"

Software development often requires:

  • Parallel work streams - Frontend & backend simultaneously
  • Dependency trees - DB schema → API → UI
  • Phased rollouts - Deploy incrementally
  • Independent reviews - Reviewable chunks
Linear AI: 15 files in one massive PR ❌
Phased approach: Phase 1 (2 files) → Phase 2 (4 files) → Phase 3 (3 files) ✅
Comparison: Linear list vs. Tree structure with phases

Pain Point #4: Context Window

The Problem: "Token limits force trade-offs"

Every AI model has a context window limit:

  • Claude 3.5 Sonnet: 200,000 tokens (~150,000 words)
  • Larger context = Higher cost
  • Larger context = Slower responses
Inefficient: 70,000 tokens (entire codebase + docs + history)
Efficient: 5,500 tokens (plan + relevant files + criteria)
Reduction: 91% fewer tokens

Token efficiency = cost efficiency + faster responses

Bar chart: Unstructured (70K) vs PARA (5.5K)

The Solution: PARA-Programming Methodology

Structured five-step workflow:

(Plan) → (Review) → (Execute) → (Summarize) → (Archive)
   ↓         ↑          ↓            ↓            ↓
 Create   Human    Implement    Document      Store
  Plan   Approve  + Commit      Results     History

The Five Steps

  1. Plan - Create structured plan before coding
  2. Review - Human validates approach
  3. Execute - Implement with git branches and atomic commits
  4. Summarize - Generate comprehensive summary from git history
  5. Archive - Preserve context for future reference
Circular workflow diagram: Plan → Review → Execute → Summarize → Archive

The Context Directory: Persistent Memory

project/
├── context/
│   ├── context.md           # Active session state
│   ├── plans/               # Pre-work planning docs
│   │   └── 2026-01-05-add-auth.md
│   ├── summaries/           # Post-work reports
│   │   └── 2026-01-05-add-auth-summary.md
│   ├── archives/            # Historical snapshots
│   ├── data/                # Input files, datasets
│   └── servers/             # MCP tool wrappers
└── CLAUDE.md                # Project-specific context

Key insight: This directory is the persistent memory layer. Files survive session boundaries.

How It Solves: Auditability ✓

Complete Decision Trail

Every PARA session produces:

  1. Plan document - Objective, approach, risks
  2. Git commits - One per completed todo, atomic changes
  3. Summary document - What changed, why, key decisions
Example Audit Trail:
Plan: "Add rate limiting using Redis"
Commits: "Add dependency" → "Create middleware" → "Apply to routes"
Summary: "Token bucket algorithm, 100 req/min limit"

How It Solves: Structure ✓

Phased Plans for Complex Work

  • Simple Plan: Single file, one branch, one PR (3-5 files)
  • Phased Plan: Master plan + phase sub-plans (>5-10 files)
Example: "Implement search with Elasticsearch"
Phase 1: ES setup + config → PR #1 → Merge
Phase 2: Indexing + API → PR #2 → Merge
Phase 3: Frontend UI → PR #3 → Merge

Each phase: Independently testable, reviewable, deployable
Tree: Master plan → 3 phases, each with branch & PR

How It Solves: Context Window ✓

Token-Efficient Workflow

  1. Plan-Scoped Context - Only load relevant files (2K-5K tokens)
  2. Preprocessing with MCP - Filter/summarize large data outside LLM
  3. Lazy Loading - Load files only when needed

Decision Guide: When to Use PARA

✅ Always Use PARA For:

  • Code changes
  • Architecture decisions
  • File modifications
  • Configuration changes
  • Database changes
  • Testing implementation

Rule: If git changes → PARA

❌ Skip PARA For:

  • Informational queries
  • Code navigation
  • Explanation requests
  • General questions
  • Quick lookups

Rule: If read-only → skip
